线程信号的目的是使不同的线程可以互发信号,此外线程信号可以让一个线程等待来自其它线程的信号

输出

这个while死循环有个专业名称叫忙碌等待,即当这个线程在等待时忙碌于查询工作

wait()/notify()/notifyAll()

java内置了使线程在等待过程中保持闲置的机制,必须和synchronized一起使用

wait

一个线程调用无论任何对象的wait()方法,都会进入闲置(睡眠)状态(释放掉锁),直到另一个线程调用之前对象的notify方法

notify

通知线程(只能通知一个),不释放锁,不存在锁竞争问题

notifyAll

通知所有wait中的线程,存在锁竞争问题

约束条件

1. 一个线程为了调用某个对象的wait()或者notify方法必须先获得这个对象的锁
也就是说调用wait或者notify方法的线程必须从一个​synchronized代码块中来调用.
2. ​注意这里使用的 synchronized 的输入参数必须是想调用 wait()，notif() 或者 notifyAll() 方法的那个对象

信号丢失问题

触发信号丢失,比如一个线程在调用wait方法之前调用notify方法

解决方法

将触发信号存储到成员变量中

实例

public class MyWaitNotify2{
MonitorObject myMonitorObject = new MonitorObject();
boolean wasSignalled = false;
public void doWait(){
synchronized(myMonitorObject){
if(!wasSignalled){
try{
myMonitorObject.wait();
} catch(InterruptedException e){...}
}
//clear signal and continue running.
wasSignalled = false;
}
}
public void doNotify(){
synchronized(myMonitorObject){
wasSignalled = true;
myMonitorObject.notify();
}
}
}

虚假醒来问题

定义

可能会因为一些莫名其妙的原因，在还没有调用 notify() 和 notifyAll() 方法时线程可能就从等待中醒了过来。这被成为虚假的醒来 —— 即无原因的醒来

解决方法

为了防止虚假醒来所带来的不利影响，触发信号变量（即例子中的 wasSignalled ）需要在一个循环中被重复检查（而不只是用 if 语句），这样的循环也被称为自旋锁（spin lock）

实例

public class MyWaitNotify3{
MonitorObject myMonitorObject = new MonitorObject();
boolean wasSignalled = false;
public void doWait(){
synchronized(myMonitorObject){
while(!wasSignalled){
try{
myMonitorObject.wait();
} catch(InterruptedException e){...}
}
//clear signal and continue running.
wasSignalled = false;
}
}
public void doNotify(){
synchronized(myMonitorObject){
wasSignalled = true;
myMonitorObject.notify();
}
}
}

多个线程等待相同信号的问题

对于多个睡眠线程等待 notifyAll 的触发信号，上面提到的使用 while 循环的方法会使得其中只有一个线程被唤醒并继续运行。这是因为只有一个线程可以获得监视对象的锁，也就会有一个线程率先被唤醒然后将 wasSignalled 置为 false。这个线程随后退出 synchronized 代码块，此时其他的线程就可以进入 doWait() 函数。然而此时，wasSignalled 已经被置为 false，所以其他的线程就会一直等待下去

不要调用常量字符串或者全局对象的wait方法

jvm会在内部自动将内容相同的String转变为相同的对象,这样就会导致出现错误唤醒问题
